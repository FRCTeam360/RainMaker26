---
name: can-id-audit
description: Scan Constants.java (and drivetrain/TunerConstants files) for two device CAN IDs sharing the same value on the same CAN bus. Use when reviewing a PR that adds/changes CAN IDs, wiring a new device, or when asked to check for CAN ID conflicts.
argument-hint: No arguments required
---

A CAN ID conflict (two devices with the same ID on the same bus) is a breaking hardware change per this repo's PR review guidelines. IDs are only meaningful *within* the same bus — different robot configs (PracticeBot, CompBot, WoodBot) run on physically separate robots and routinely reuse the same numeric IDs on purpose, so never flag a match across different robot config classes as a conflict.

## Steps

### 1. Collect ID declarations, grouped by scope

Grep `src/main/java/frc/robot/Constants.java` for `_ID = <int>` and `CANBus(...)` declarations (pattern: `Grep -n "_ID = |CANBus\("`). Group each ID by its enclosing static nested class (e.g. `PracticeBotConstants`, `CompBotConstants`, `WoodBotConstants`, `SimulationConstants`) — that nested class plus its `CANBUS` field (or `Constants.RIO_CANBUS` for PDH-type devices) defines the bus scope.

Also check drivetrain CAN IDs: search for a `TunerConstants*.java` file (generated by Phoenix Tuner X for swerve modules) under `src/main/java`. These IDs live on the same physical bus as the matching robot config's other devices but in a separate generated file — include them in that config's scope if found.

### 2. Check for duplicates within each bus scope

Within each (nested class, bus) group, list every `(constant name, ID value)` pair. Flag any ID value that appears more than once within the same group — that's a real conflict, since two physical devices cannot share one CAN ID on one bus.

Do not flag:
- The same ID value appearing in different robot config classes (e.g. `PracticeBotConstants.HOOD_ID` vs `CompBotConstants.HOOD_ID` sharing a number) — different physical robots, not a conflict.
- Commented-out ID declarations (e.g. `// public static final int HOPPER_SENSOR_ID = 25;`) — not live code.

### 3. Report

For each real conflict found: name both constants, their shared ID value, the file, and line numbers. If a `TunerConstants` file's module IDs weren't found or couldn't be parsed, say so explicitly rather than silently skipping — the user should know that scope wasn't checked.

## Output

If no conflicts: state plainly that no CAN ID conflicts were found, and which scopes were checked (list the robot config classes/buses covered).

If conflicts are found: list each one as `<ConstantA> and <ConstantB> both use ID <N> on <bus/config>` with file:line references. Do not edit `Constants.java` automatically — flag for the user to resolve, since picking the correct new ID requires knowing which physical device is actually free.
